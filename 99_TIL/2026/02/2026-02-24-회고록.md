### 2. SimMonitor (feature/yoon 브랜치) — UUID 분석 ⭐

- UUID가 **WCF MessageID**임을 파악
- 흐름 분석: 추출 → RaiseEvent → SimprecClient / WebSocket / MonitorService 3경로
- **문제 발견**: 개별 이벤트 UUID가 eth-simprec에서 버려지고 있음
- **중요 사실**: UUID는 이벤트별 고유값이 아님 (1 블록 = 다수 이벤트) → UNIQUE 제약조건 사용 금지

### 4. 검증: QCPR/CLINICAL 중복 및 CPR EVENT 누락

#### 문제 1: QCPR/CLINICAL 중복 출력 — 해결 확인
- `wcf_events` 로그에서 QCPR 파라미터가 **QCPR 타입으로만** 출력됨
- CLINICAL 타입으로 QCPR 데이터가 중복 전송되지 않음
- `ws_broadcast` 로그에서도 `eventType:"QCPR"`으로만 브로드캐스트, 단위(mm, cpm 등) 포함 확인

#### 문제 2: CPR EVENT 누락 — 해결 확인
- `LOGELEMENT_SKIP`은 Pattern A(벌크 히스토리)의 이전 세션 데이터에서만 발생
- Pattern B(실시간 GString)의 CPR 이벤트는 모두 정상 처리:

| 시간 | 이벤트 | LogElement Id |
|---|---|---|
| 00:57.1 | 압박 시작 | Id=10 |
| 00:57.1 | CPR 시작 | Id=11 |
| 01:16.9 | 압박 중단 | Id=33 |
| 01:16.9 | CPR 중지 | Id=34 |

- `ws_broadcast`에서 4개 CPR 이벤트(COMPRESSION_START, CPR_START, COMPRESSION_STOP, CPR_STOP) 모두 WebSocket 브로드캐스트 확인
- 커밋 `e731de2`로 feature/yoon 브랜치에 푸시 완료

## 💡 핵심 교훈
1. **UUID ≠ 이벤트 고유값**: WCF MessageID 기반이라 1블록에 여러 이벤트가 같은 UUID를 공유 — UNIQUE 제약조건을 걸면 안 됨
2. **중간 접속 문제의 정석 해결**: WebSocket 연결 시 상태 → 세션 → 바이탈 → 이벤트 순서로 초기 데이터를 밀어주는 것이 핵심
3. **Pattern A vs B 분리의 효과**: 벌크 히스토리(Pattern A)와 실시간 이벤트(Pattern B)를 분리한 덕분에 CPR 이벤트 누락 없이 깔끔하게 처리
