# 2026년 2월 3일 (월) - WCF 이벤트 지연 버그의 근본 원인 해결

## 🎯 주요 성과
- WCF 이벤트 ~90초 지연 문제의 실제 원인 파악 및 해결
- FileInfo 캐싱 버그 수정으로 파일 감지 지연 90초 → 4초로 대폭 개선
- 디버그 로깅 시스템 개선으로 향후 문제 추적 능력 강화

## 📋 작업 내용

### 1. WCF 이벤트 지연 문제 심층 분석
- **문제 상황**: WCF 이벤트가 SimMonitor와 SIMPREC에 약 90초 지연되어 표시
- **분석 방법**: 원본 LLEAP WCF 로그(330MB) 타임스탬프 상세 분석
- **핵심 발견**: 지연의 원인이 LLEAP이 아닌 SimMonitor 자체에 있음을 확인

### 2. 근본 원인 규명
- **버그 위치**: `WcfEventCapture.cs`의 FileInfo.Length 사용 부분
- **문제점**: FileInfo.Length가 캐시된 값을 반환하여 실시간 파일 변경 감지 실패
- **영향 범위**: 5개 주요 메서드에서 동일한 패턴 발견

### 3. 코드 수정 및 개선
```csharp
// 이전 (버그): FileInfo.Length는 캐시된 값 반환
var info = new FileInfo(wcfLogPath);
if (info.Length <= wcfPosition) return;

// 수정 후: FileStream.Length로 실시간 파일 크기 확인
using (var fs = new FileStream(wcfLogPath, FileMode.Open, FileAccess.Read, FileShare.ReadWrite))
{
    long currentLength = fs.Length;  // 실시간 크기!
    if (currentLength <= wcfPosition) return;
}
```

### 4. 수정된 메서드 목록
| 메서드 | 변경 내용 |
|---|---|
| `Start()` | 초기 파일 크기 읽기 시 FileStream 사용 |
| `MonitorLoop()` | 5초 상태 로깅 시 FileStream 사용 |
| `CheckWcfLog()` | 핵심! 파일 변경 감지 시 FileStream 사용 |
| `ManageWcfLogOnStartup()` | 로그 크기 확인 시 FileStream 사용 |
| `CheckWcfLogSizeRuntime()` | 런타임 크기 확인 시 FileStream 사용 |

### 5. 디버그 로깅 시스템 개선
- **[WCF-DEBUG]** 메시지를 UI와 wcf_raw 파일 양쪽에 기록
- **[WCF-ERROR]** 레이블로 에러도 파일에 영구 기록
- 향후 문제 추적 및 디버깅 효율성 대폭 향상

### 6. 테스트 및 검증
- **개선 결과**: SimMonitor 파일 감지 지연 90초 → 4초로 단축 (95% 개선)
- **안정성 테스트**: 장시간 실행 시에도 안정적으로 동작 확인
- **로그 무결성**: 모든 WCF 이벤트가 누락 없이 정상 기록됨

## 💡 핵심 교훈
1. **성능 문제의 원인은 예상 밖의 곳에**: LLEAP이 아닌 SimMonitor 자체의 버그였음
2. **FileInfo vs FileStream**: .NET의 FileInfo는 성능을 위해 캐싱하므로, 실시간 모니터링에는 부적합
3. **로깅의 중요성**: 디버그 로그를 파일에도 기록하여 사후 분석 능력 확보
