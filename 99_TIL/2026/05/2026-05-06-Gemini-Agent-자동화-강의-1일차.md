## 1교시 - AI 도구 활용 사례 & 핵심 개념

### NotebookLM으로 행사 전문가 만들기
Google Cloud Next 행사에서 100개가 넘는 발표자료들을 전부 NotebookLM 소스로 넣어두고, 이 노트북을 "행사 전문가 AI"로 만들어서 수많은 발표 내용을 한꺼번에 정리하고 질문에 답하게 함. 자료를 일일이 읽지 않아도 AI가 맥락을 파악해서 요약해주는 Agentic AI 활용 사례.

### Agentspace - 내 업무 회고를 AI가 대신
"Jerry가 지난 3개월 동안 한 일은?" 이라고 질문하면, Jerry Todo Agent가 연결된 Google Doc(할 일 목록, 업무 기록)을 참고해서 Q3 업무 요약을 자동 생성해줌. 내가 기억에 의존해서 쓰는 게 아니라 객관적인 데이터 기반으로 정리해주기 때문에, 주간보고 / 월간회고 / 인사평가 자료 작성이나 면접에서 "지난 프로젝트에서 뭐 했냐"는 질문 대응에도 활용 가능.

### Knowledge Graph
데이터를 단순히 저장하는 게 아니라 개체들 간의 관계를 연결해서 표현하는 구조. AI Agent가 단순 검색이 아닌 맥락을 이해하고 추론할 수 있는 이유가 이 Knowledge Graph 덕분. 강사가 Palantir 주가가 비싼 이유로 언급한 Ontology도 같은 개념으로, 기업 내 데이터들의 관계를 정의해두는 것. 복잡하고 반복적인 업무일수록 이 구조로 자동화 가능성이 높아짐.

### AI 활용 평가 방식 - 판단력이 핵심
강원대 수업 사례. AI 사용은 전부 허용, 발표자료도 레포트도 AI로 만들어도 됨. 단, 자료에 대한 이해와 책임은 전적으로 학생 본인이 짐. "이 내용을 이해했냐, 못 했냐"를 중요시하고, 정확하지 않은 자료를 냈을 때 책임도 학생이 짐. AI가 뽑아준 결과물을 판단하고 검증하는 능력이 핵심. 앞으로 실무에서도 동일하게 적용되는 원칙.

### 경력같은 신입 - 포트폴리오가 전부
어떤 학생이 이력서를 넣고 면접을 약 100번 다녔다고 함. IT 자격증 같은 스펙보다 포트폴리오가 훨씬 중요하다는 게 요점. 학교 팀프로젝트 수준이 아니라, 본인 프로젝트를 6개월 동안 리팩토링하면서 스스로 개선한 걸 포트폴리오로 제출함. 리팩토링 과정에서 코드를 다시 이해하고, 왜 이렇게 짰는지 파악하면서 성장 - 이걸 강사는 **역방향 배움(Reverse Learning)** 이라고 표현. 결과적으로 면접관이 경력자인 줄 알았다고 함.

### 온보딩 2.0 - 바이브코딩이 바꾼 신입의 포지션
기존엔 주니어가 노션 문서 지옥을 읽으며 혼자 파악하고, 열심히 개발해서 가져가도 기술 스택 통합이 안 되고, CTO는 "문서에 다 있는데 왜 모르냐", CEO는 "저 신입 괜찮은 거냐" 의심하는 구조였음. 이제는 Gemini CLI 같은 AI Agent로 코드 자체를 분석하고, 통합 환경도 AI가 도와주면서 주니어가 훨씬 빠르게 팀에 기여 가능해짐. **AI는 주니어와 시니어 사이의 완벽한 미들웨어.** 강사는 "CEO한테 AI를 직접 가르치게 하는 게 가장 좋은 피드백"이라고도 했는데, 가르치면서 본인이 뭘 모르는지 드러나고 실제 업무 문제도 나오기 때문.

## 2교시 - NotebookLM 실습 & Gems & 프롬프트 전략

### AI에게 비판적 검토 요청하기
AI는 기본적으로 사용자 말에 동조하는 경향이 있음. 프롬프트에 "비판적으로 검토해줘" + "네 의견은 네 의견이라고 명시해줘" 두 가지를 습관적으로 추가하는 걸 추천. AI 아첨을 필터링하는 실용적인 팁.

### NotebookLM 킬러 기능 - 스토리가 먼저, 슬라이드는 나중
강사가 킬러 기능으로 꼽은 오디오 오버뷰와 슬라이드 생성. 핵심 워크플로우는 다음과 같다:

**말(녹음) → 텍스트 변환 → NotebookLM 소스 업로드 → 슬라이드 생성 → 오디오 대본 생성 → 발표**

보통 사람들이 슬라이드 틀 먼저 만들고 내용 채우다가 하고 싶은 말을 못 담는 경우가 많은데, 이 방식은 반대로 재밌는 주제로 말을 먼저 다 쏟아내고 그걸 텍스트로 변환해서 슬라이드를 만드는 것. 그래야 내가 전하고 싶은 스토리를 100% 담을 수 있다. 이 루프는 무한으로 돌아감 - 대본이 다시 소스가 돼서 또 다른 자료 생성 가능.

스튜디오 기능: 슬라이드, 오디오 오버뷰, 보고서, 플래시카드, 퀴즈, 인포그래픽, 데이터 표 생성 가능.

### Gems = Claude Projects와 같은 개념
자주 쓰는 구조화된 프롬프트를 미리 작성해서 재사용하는 개인화 Agent. Instructions를 잘 써야 Gem도 똑똑해짐. Instructions 작성할 때 포인트:
- AI한테 직접 "이런 Gem 만들고 싶은데 Instructions 짜줘" 하고 시키는 게 퀄리티가 높음
- 만족할 때까지 티키타카로 다듬는 것. 한번에 나오는 건 없음
- **프롬프트 4단계 공식**: Role(역할/페르소나) → Context & Goal(배경과 목적) → Format & Constraints(형식과 제약조건) → Feedback(반복 피드백)

### Google Opal & 워크플로우
n8n의 단순화 버전. "트리거 → 작업 → 결과" 흐름을 노코드로 연결하는 워크플로우 도구. Claude에서는 MCP(Model Context Protocol)로 외부 서비스와 연결하는 게 같은 개념.

## 3교시 - LLM 개념 & AI 트렌드

### LLM의 본질 - 컨텍스트 윈도우와 기억
LLM은 진짜 "기억"이 없고, 대화 내용을 통째로 들고 있다가 필요할 때 뒤져서 답하는 구조 - **커닝**. 대화가 길어져서 컨텍스트 윈도우 밖으로 밀려나면 그냥 잊어버림. 강사 표현으로는 **"얼어있는 뇌를 매번 해동시켜서 쓰는 것"**. 인간은 경험하면서 실시간으로 지속학습하지만, LLM은 학습(Training)이 끝나면 고정되고 이후엔 추론(Inference)만 함. 그래서 cut-off date 이후 정보는 모르고, RAG나 검색 연동으로 보완함.

입력 토큰은 크고(수십만~100만) 출력 토큰은 작음(수만). 이 구조 때문에 AI가 축약을 잘하는 게 구조적으로 당연한 것.

### Predictive AI vs Generative AI
- **Predictive AI**: 특정 목적의 레이블링 데이터 수천~수만 개 필요. 바닥부터 학습. 막대한 데이터 구축 비용.
- **Generative AI**: 2~3개 예시로 충분. 세상의 지식을 사전 학습(Pre-trained)한 상태. 프롬프트 설계에 따른 다목적 활용.

> **"데이터가 없어서 AI를 못 한다는 변명은 끝났다. 이제는 프롬프트와 맥락(Context) 설계의 싸움이다."**

### NotebookLM으로 강의 슬라이드 자동 생성 실습
오늘 강의 PDF를 NotebookLM에 넣고 슬라이드를 생성하니 15장 분량의 깔끔한 발표자료가 자동으로 만들어짐. 1교시에 배운 워크플로우를 수업 중에 직접 실증.

## 4교시 - LLM 특징 심화 & 바이브 코딩 실습

### LLM의 특징 정리
- **학습 불필요**: 데이터 거의 불필요. 개/고양이 사진 1000장이 아니라 2~3장으로 충분
- **85% 업무 커버**: 프롬프트로 잘 구슬리면 못 하는 게 없음. 나머지 15%는 추천 모델처럼 전문 학습 필요
- **ROI가 핵심**: 단순 반복 작업엔 LLM이 오히려 비용 낭비일 수 있음. 결국 비용 대비 효과로 판단
- **Hallucination**: 완전 제거 불가. Grounding / Fact Check / Data Cleansing / Evaluation으로 관리하는 게 현실적. Zero에 가깝게 줄일수록 비용이 급격히 증가
- **Non-deterministic**: 같은 질문해도 답이 매번 다름. LLM의 본질적 특성

### AI Agent 행동 강령 (CRITICAL CORE DIRECTIVES)
AI 코딩 에이전트한테 줄 수 있는 필수 규칙 모음. 핵심은 "멋대로 하지 말고, 모르면 물어보고, 하기 전에 말해라":
- **No Silent Mutations**: 파일 수정/삭제 전 반드시 목적과 이유 먼저 설명
- **No Guessing**: 모호하면 임의로 추측하지 말고 반드시 먼저 질문
- **No Wait After Explain**: 설명 끝나면 바로 도구 호출, 대답 기다리지 말 것
- **Atomic Workflow**: 복잡한 작업은 단계 쪼개서 진행, 수정 후 자가 검증
- **Pre-Delete Check**: 요소 삭제/이동 전 호출자(Callers) 파악 필수

Gems나 Claude Projects Instructions 만들 때도 이런 식으로 행동 규칙을 구체적으로 써주는 게 훨씬 안전하고 퀄리티 높음.

### Skill & Conductor - 컨텍스트를 파일로 관리
세션이 끊겨도 이어서 작업 가능하게 하는 핵심 개념. 프로젝트 루트에 `CLAUDE.md` 같은 파일을 만들어두면 새 세션 시작할 때 자동으로 읽어서 프로젝트 맥락을 파악하고 바로 이어서 코딩 가능. **AI의 기억은 없고, 컨텍스트만 있다. 그 컨텍스트를 얼마나 잘 설계해두냐가 AI를 잘 쓰는 핵심.**

### 바이브 코딩 실습 - Gemini Canvas
Gemini Canvas에서 혈당 관리 CSV 데이터 + 프롬프트 한 방에 → 인터랙티브 꺾은선 그래프 앱 즉시 생성. 마우스 포인팅, 메모 추가, CSV 업로드 기능까지 포함. 코드 한 줄 안 써도 됨.

### PRD 활용법 - 아이디어를 실행으로
"이걸 PRD로 만들어줘" 한 마디로 요구사항 정의서 → 기술 스펙 → 구현 단계까지 자동 생성. 활용 전략:

**아이디어 → PRD 생성 → 첫 답변 저장 → 항목 하나씩 도장깨기 → 완성**

한 번에 전부 구현하려 하면 산으로 가고, PRD를 로드맵으로 삼아서 하나씩 티키타카로 구현하면 완성도가 올라감.

## 오늘의 핵심 인사이트

1. **판단력이 전부다** - AI가 만든 결과물에 대한 이해와 책임은 인간이 짐. AI 출력을 검증하고 판단하는 능력이 앞으로의 핵심 역량.
2. **스토리가 먼저, 슬라이드는 나중** - 말로 먼저 쏟아내고 AI가 구조화. 하고 싶은 말을 100% 담는 방법.
3. **AI의 기억은 없고 컨텍스트만 있다** - 컨텍스트 설계가 AI 활용의 핵심. CLAUDE.md, Instructions, NotebookLM 소스가 모두 같은 맥락.
4. **개발자의 역할은 코더에서 아키텍트로** - What & Why에 집중하고 How는 AI에게. 바이브 코딩 생산성은 최소 10배.
5. **Embrace Vibe Coding → Master the Workflow → Stay Human** - 성과와 속도는 AI에게, 가치와 책임은 인간이 주도.
