# SIMPREC 동작 원리 — 면접용 설명 가이드

> 코드를 보지 않고도 SIMPREC 시스템의 핵심 로직을 남에게 설명할 수 있도록 정리한 문서입니다.

---

## 1. 전체 시스템을 한 문장으로

> "의료 시뮬레이션 수업에서 **LLEAP 시뮬레이터가 시나리오를 시작하면 자동으로 멀티 카메라 녹화가 시작**되고, 녹화 중 발생한 **바이탈 사인과 임상 이벤트가 영상과 함께 기록**되며, 수업 후 **학생이 웹에서 영상을 보고 자기 성찰 보고서를 제출**하는 플랫폼."

---

## 2. 5단계 동작 원리

### 전체 흐름 한눈에 보기

```
교수자가 수업 일정/학생 명단을 미리 등록
    │
    ▼
[1단계] LLEAP에서 시나리오 시작 → SimMonitor가 감지
    │
    ▼
[2단계] SIMPREC이 자동으로 멀티 카메라 녹화 시작
    │
    ▼
[3단계] 실시간으로 바이탈/이벤트 기록 + 모니터링 화면 표시
    │
    ▼
[4단계] 시나리오 종료 → 녹화 자동 중지 + 영상 저장
    │
    ▼
[5단계] 학생에게 이메일 발송 → 영상 시청 → 보고서 제출
```

---

### [1단계] 시나리오 시작 감지 + 학생 자동 매칭

**비유:** 식당 대기줄 — 다음 손님이 자동으로 호출되는 것

**동작 원리:**

교수자가 미리 수업 일정을 등록해 둡니다: "10시 홍길동, 10시20분 김철수, 10시40분 이영희"

LLEAP 시뮬레이터에서 시나리오를 시작하면, SimMonitor(미들웨어)가 이를 감지하여 SIMPREC 서버에 "시나리오 시작됐다"라고 알려줍니다.

SIMPREC은 **스케줄 큐에서 다음 대기 학생을 자동으로 찾아** 이 세션에 연결합니다.

```
스케줄 큐:
  1번 홍길동 [완료] ← 이미 끝남
  2번 김철수 [대기] ← 이번에 자동 매칭!
  3번 이영희 [대기]

LLEAP 시나리오 시작
  → SimMonitor → SIMPREC "시나리오 시작"
  → SIMPREC: "다음 대기 학생은 김철수" → 자동 매칭
```

**왜 중요한가?**
- 교수자가 매번 학생 이름을 선택하거나 입력할 필요 없음
- 순서를 잘못 선택하는 실수 방지
- 하루 20+ 세션 운영 시 엄청난 시간 절약

---

### [2단계] 멀티 카메라 자동 녹화

**비유:** 영화 촬영장의 "액션!" 사인 — 모든 카메라가 동시에 녹화 시작

**동작 원리:**

SIMPREC이 "녹화 시작" 명령을 내리면, **MEMORA V2**라는 전용 레코더가 모든 카메라를 **동시에** 녹화합니다.

```
SIMPREC 서버: "녹화 시작!"
    │
    ├──→ 카메라 1 (정면): MEMORA가 RTSP 스트림을 MP4로 저장
    ├──→ 카메라 2 (측면): MEMORA가 RTSP 스트림을 MP4로 저장
    └──→ 카메라 3 (상단): MEMORA가 RTSP 스트림을 MP4로 저장
         │
         └── Barrier Sync: 모든 카메라의 시작 시점을 맞춤
             (오차 0.0001초 = 100마이크로초 이내)
```

**Barrier Sync란?**

카메라마다 네트워크 지연이 조금씩 다르기 때문에, 단순히 "동시에 녹화 시작"하면 미세하게 어긋납니다.

MEMORA는 모든 카메라가 **키프레임(영상의 기준점)을 맞출 때까지 대기**한 뒤 동시에 녹화를 시작합니다. 이 과정을 Barrier Sync라고 합니다.

```
카메라 1: ──────[키프레임 도착]──── 대기 중...
카메라 2: ────────[키프레임 도착]── 대기 중...
카메라 3: ──[키프레임 도착]──────── 대기 중...
                                      │
                                      ▼ 모두 준비됨!
                               동시에 녹화 시작 ▶
```

**Warm Buffer(10초 프리롤):**
MEMORA는 녹화 명령 전부터 10초간 영상을 메모리에 보관합니다. 이를 통해 녹화 시작 직전의 영상도 놓치지 않습니다.

---

### [3단계] 실시간 바이탈/이벤트 기록

**비유:** 블랙박스 — 영상뿐 아니라 당시 속도/GPS 데이터도 함께 기록하는 것

**동작 원리:**

녹화가 진행되는 동안, SimMonitor가 시뮬레이터의 바이탈과 이벤트를 실시간으로 SIMPREC에 전송합니다.

```
SimMonitor ──WebSocket(초당 20회)──→ SIMPREC 서버
    │                                    │
    │  보내는 데이터:                     │  하는 일:
    │  - 심박수 (HR)                     │  - DB에 저장 (session_vitals)
    │  - 혈압 (BP)                       │  - DB에 저장 (session_events)
    │  - 산소포화도 (SpO2)               │  - 웹 대시보드에 실시간 전송
    │  - 호흡수 (RR)                     │  - 이벤트 로그 업데이트
    │  - CPR 시작/중지                   │
    │  - 약물 투여                       │
    │  - 상태 변화                       │
    │                                    ▼
    │                             교수자 모니터링 화면
    │                             (WebRTC 영상 + 바이탈 + 이벤트)
```

**바이탈 저장 빈도:**
- SimMonitor → SIMPREC: **초당 20회** (20Hz) — 실시간 표시용
- SIMPREC → DB: **초당 1회** (1Hz) — 저장 효율화 (20배 압축)

왜 전부 저장하지 않나?
- 20Hz로 1시간 녹화하면 72,000개 레코드 → DB 부담
- 1Hz로 저장해도 시뮬레이션 디브리핑에는 충분 (1초 단위 변화 추적)

---

### [4단계] 녹화 종료 + 영상 저장

**비유:** 영화 촬영장의 "컷!" 사인

**동작 원리:**

LLEAP에서 시나리오가 종료되면 SimMonitor가 SIMPREC에 알려주고, SIMPREC은 녹화를 중지합니다.

```
LLEAP 시나리오 종료
  → SimMonitor → SIMPREC "세션 종료"
  → MEMORA에 KILL 명령 → 각 카메라 MP4 파일 마무리
  → 파일 메타데이터 기록:
      카메라 1: duration=1800초, sync_offset=+0μs (앵커)
      카메라 2: duration=1800초, sync_offset=+45μs
      카메라 3: duration=1800초, sync_offset=-12μs
  → 스케줄 큐 업데이트: 김철수 [완료]
  → 프론트엔드에 "녹화 완료" 알림
```

**동기화 오프셋이란?**

카메라 1을 기준(앵커)으로, 카메라 2는 +45마이크로초, 카메라 3는 -12마이크로초 차이가 났다는 의미입니다. 이 값을 저장해 두면 **재생할 때 정확히 보정**할 수 있습니다.

---

### [5단계] 학생 영상 배포 + 보고서 수집

**비유:** 온라인 시험 — 링크를 받아 접속하고, 문제에 답변을 제출하는 것

**동작 원리:**

```
교수자가 "이메일 발송" 클릭
  → 학생별 고유 토큰이 포함된 접속 링크 생성
  → 이메일 자동 발송

학생이 링크 클릭
  → 토큰 검증 → 자기 세션의 녹화 영상만 접근 가능
  → 멀티 카메라 동기화 재생 (카메라 전환 가능)
  → 영상 시청 후 자기 성찰 보고서 작성/제출

교수자가 관리 페이지에서
  → 학생별 제출 여부 확인
  → 보고서 내용 열람/평가
```

## 3. 핵심 기술 로직 상세

### 3.1 스케줄 큐 매칭 — "다음 학생은 누구?"

SIMPREC의 가장 핵심적인 자동화 로직입니다.

```
교수자가 미리 등록한 스케줄:
┌─────┬────────┬───────────┬──────────┐
│ 순서 │ 학생   │ 예정 시간  │ 상태     │
├─────┼────────┼───────────┼──────────┤
│  1  │ 홍길동 │ 10:00     │ 완료 ✅  │
│  2  │ 김철수 │ 10:20     │ 대기 ⏳  │  ← 다음은 이 학생!
│  3  │ 이영희 │ 10:40     │ 대기 ⏳  │
│  4  │ 박민수 │ 11:00     │ 대기 ⏳  │
└─────┴────────┴───────────┴──────────┘

LLEAP에서 시나리오 시작 신호가 오면:
1. 상태가 "대기"인 스케줄 중 가장 앞 순서를 찾음
2. 해당 학생 정보를 세션에 연결
3. 스케줄 상태를 "진행 중"으로 변경
4. 녹화 시작

시나리오 종료 시:
1. 스케줄 상태를 "완료"로 변경
2. 다음 대기 학생 정보를 SimMonitor에 반환
   → SimMonitor 오버레이에 "다음: 이영희" 표시
```

**왜 시간 기반이 아니라 순서 기반인가?**
- 시뮬레이션은 예정 시간대로 진행되지 않음 (시나리오 길이 가변)
- 10:00에 시작한 세션이 10:35에 끝날 수 있음
- 시간 기반이면 10:20 학생을 건너뛸 위험
- 순서 기반이면 **항상 다음 대기 학생에게 정확히 매칭**

---

### 3.2 Barrier Sync — "카메라가 동시에 녹화되는 원리"

멀티 카메라 녹화에서 가장 기술적으로 어려운 부분입니다.

```
문제: 각 카메라의 RTSP 스트림은 네트워크 지연이 다름
  카메라 1: ─────[프레임 도착]──────────────
  카메라 2: ──────────[프레임 도착]──────────
  카메라 3: ───[프레임 도착]────────────────

  → 그냥 녹화하면 카메라별로 시작 시점이 다름
  → 재생 시 "같은 장면인데 카메라마다 다른 시간"이 됨

해결: Barrier Sync (장벽 동기화)

  1단계: 모든 카메라의 영상을 메모리에 버퍼링 (아직 저장 안 함)

  2단계: 각 카메라에서 "키프레임"(기준 프레임)이 도착할 때까지 대기
         카메라 1: 키프레임 도착! → 장벽에 도착 ✅
         카메라 2: 아직...         → 대기 중 ⏳
         카메라 3: 키프레임 도착! → 장벽에 도착 ✅
         카메라 2: 키프레임 도착! → 장벽에 도착 ✅

  3단계: 모든 카메라가 장벽에 도착 → 동시에 녹화 시작!
         → 각 카메라의 시작 시점 차이를 마이크로초 단위로 기록
         → 이 차이를 "sync_offset_us"로 DB에 저장

  4단계: 재생 시 오프셋만큼 보정
         카메라 1: 재생 위치 = 현재 시간 + 0μs (앵커)
         카메라 2: 재생 위치 = 현재 시간 + 45μs
         카메라 3: 재생 위치 = 현재 시간 - 12μs
```

### 3.3 이벤트 중복 제거 — "같은 이벤트가 2번 표시되는 문제"

```
문제:
  SimMonitor → SIMPREC으로 이벤트가 2가지 경로로 도착:

  경로 1: HTTP POST /events    → DB 저장 + 프론트엔드 전송
  경로 2: WebSocket wcfEvent   → 프론트엔드 직접 전송

  → 프론트엔드에서 같은 "CPR 시작" 이벤트가 2번 표시됨

해결:
  1. 모든 이벤트에 고유 ID(UUID) 부여
  2. 프론트엔드에서 이미 표시한 UUID는 무시
  3. 백엔드에서도 같은 UUID의 이벤트는 DB에 1번만 저장

  이벤트 흐름:
  SimMonitor: "CPR 시작" (uuid: abc123)
    ├── HTTP /events → DB 저장 (uuid: abc123)
    └── WebSocket    → 프론트엔드 수신 (uuid: abc123)
                       → "abc123은 이미 표시함 → 무시"
```

### 3.4 WebSocket 중간 접속 — "늦게 켜도 현재 상태가 보이는 원리"

```
문제:
  시뮬레이션이 10분째 진행 중인데, 교수자가 이제서야 모니터링 화면을 열었음
  → WebSocket은 "연결 이후" 데이터만 전송
  → 10분간의 바이탈/이벤트를 모르니 빈 화면

해결:
  WebSocket 연결 시 서버가 "현재 상태 요약"을 즉시 전송

  1. 연결 즉시 전송하는 데이터:
     ├── 현재 바이탈 스냅샷: {HR: 120, SpO2: 98, BP: 120/80, ...}
     ├── 최근 이벤트 100개: [{CPR 시작, 10:05}, {약물 투여, 10:08}, ...]
     ├── 녹화 상태: {is_recording: true, duration: 600초}
     └── 세션 정보: {scenario: "천식 시나리오", student: "김철수"}

  2. 이후부터는 실시간 데이터 수신 (20Hz)

  결과: 교수자가 언제 화면을 열어도 즉시 현재 상태 확인 가능
```

## 4. SIMPREC과 SimMonitor의 관계

```
두 시스템은 역할이 완전히 다릅니다:

┌─────────────────────────────────────────────────┐
│ SimMonitor (미들웨어)                            │
│                                                  │
│ 역할: LLEAP 시뮬레이터의 데이터를 "추출"         │
│                                                  │
│ - LLEAP WCF 로그 파일을 실시간 파싱              │
│ - 바이탈 사인, 임상 이벤트, 시나리오 상태 추출   │
│ - 추출한 데이터를 SIMPREC에 전달                 │
│                                                  │
│ 기술: C#, .NET, WPF, Win32 API                   │
│ 위치: LLEAP가 설치된 Windows PC에서 실행          │
└───────────────────┬─────────────────────────────┘
                    │  HTTP + WebSocket
                    ▼
┌─────────────────────────────────────────────────┐
│ SIMPREC (플랫폼)                                 │
│                                                  │
│ 역할: 녹화를 "관리"하고 학생에게 "배포"          │
│                                                  │
│ - SimMonitor 데이터를 받아서 DB에 저장           │
│ - MEMORA 레코더에 녹화 시작/종료 명령            │
│ - 녹화 영상 + 바이탈 + 이벤트 통합 관리         │
│ - 학생 영상 배포, 보고서 수집, 이메일 발송       │
│ - 실시간 모니터링 대시보드 제공                  │
│                                                  │
│ 기술: React, FastAPI, MySQL, Docker              │
│ 위치: 서버 (Docker Compose로 운영)               │
└─────────────────────────────────────────────────┘
```

## 5. 면접 대비 — 예상 질문과 답변

### Q: "SIMPREC 프로젝트를 한마디로 설명해 주세요."

> 의료 시뮬레이션 수업에서 LLEAP 시뮬레이터의 시나리오 시작/종료를 자동 감지하여 멀티 카메라 녹화를 트리거하고, 녹화 중 발생하는 바이탈 사인과 임상 이벤트를 영상과 함께 통합 기록하며, 수업 후 학생에게 영상을 배포하고 자기 성찰 보고서를 수집하는 웹 기반 플랫폼입니다.

### Q: "이 프로젝트에서 본인이 담당한 역할은?"

> 프론트엔드 전체와 백엔드 핵심 기능을 담당했습니다. 구체적으로는 React 19 기반 관리자 대시보드와 학생 페이지 50+개 컴포넌트 구현, 실시간 모니터링 화면(WebRTC 영상 + WebSocket 바이탈), 멀티 카메라 동기화 재생 플레이어, 학생 학습 관리(보고서 제출/검토), Docker Compose 기반 6개 서비스 인프라 구축을 했습니다. 백엔드에서는 SimMonitor 연동 안정성(이벤트 중복 제거, 고스트 이벤트 수정, WebSocket 중간 접속), 바이탈 데이터 DB 연동, 이벤트 UUID 추적 체계를 구현했습니다.

### Q: "멀티 카메라 동기화는 어떻게 구현했나요?"

> 녹화 시 MEMORA V2 레코더가 Barrier Sync 방식을 사용합니다. 모든 카메라의 키프레임이 도착할 때까지 대기한 뒤 동시에 녹화를 시작하고, 각 카메라의 시작 시점 차이를 마이크로초 단위로 DB에 기록합니다. 재생 시에는 프론트엔드의 SyncVideoPlayer가 이 오프셋 값을 활용하여 앵커 카메라 기준으로 각 카메라의 재생 위치를 보정합니다. 이렇게 하면 50ms 이내의 동기화 정밀도를 달성할 수 있습니다.

### Q: "실시간 데이터는 어떻게 처리하나요?"

> SimMonitor에서 WebSocket으로 초당 20회 바이탈 데이터를 수신합니다. 이 데이터는 인메모리 상태(AppState)에 즉시 반영되어 프론트엔드 WebSocket 클라이언트들에게 브로드캐스트됩니다. DB 저장은 초당 1회로 스로틀링하여 부하를 줄입니다. 프론트엔드에서 모니터링 페이지를 늦게 열어도, WebSocket 연결 시 서버가 현재 바이탈 스냅샷 + 최근 100개 이벤트를 즉시 전송하므로 빈 화면 문제가 없습니다.

### Q: "SimMonitor와 SIMPREC의 차이점은?"

> SimMonitor는 LLEAP 시뮬레이터의 WCF 로그를 파싱하여 바이탈과 이벤트를 추출하는 C# 미들웨어이고, SIMPREC은 이 데이터를 받아서 녹화를 제어하고 학생에게 배포하는 웹 플랫폼입니다. SimMonitor는 "데이터 추출"에 집중하고, SIMPREC은 "녹화 관리와 학습 프로세스"에 집중합니다. 두 시스템은 HTTP REST API와 WebSocket으로 통신합니다.

### Q: "가장 어려웠던 기술적 문제는?"

> WebSocket 중간 접속 문제와 이벤트 중복이었습니다. 시뮬레이션이 진행 중인데 모니터링 화면을 열면 이전 데이터를 알 수 없어 빈 화면이 나왔습니다. WebSocket 연결 시 서버 인메모리 상태에서 현재 스냅샷을 즉시 전송하는 프로토콜을 설계해서 해결했습니다. 이벤트 중복은 HTTP와 WebSocket 양쪽으로 같은 이벤트가 도착하는 문제인데, UUID 기반 중복 제거와 세션 경계에서의 이벤트 히스토리 초기화로 해결했습니다.
