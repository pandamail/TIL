# SimMonitor 동작 원리 — 면접용 설명 가이드

> 코드를 보지 않고도 SimMonitor의 핵심 로직을 남에게 설명할 수 있도록 정리한 문서입니다.

---

## 1. 전체 동작을 한 문장으로

> "LLEAP 시뮬레이터가 남기는 **로그 파일을 실시간으로 감시**해서, 시나리오 시작/종료·약물 투여·CPR 같은 **임상 이벤트를 자동 추출**하고, 녹화 시스템(SIMPREC)에 **자동으로 전달**하는 미들웨어."

---

## 2. 4단계 동작 원리

### 전체 흐름 한눈에 보기

```
LLEAP 시뮬레이터
    │
    │  내부 통신 내용을 로그 파일에 기록
    ▼
[1단계] 로그 파일 감시
    │  "파일에 새 내용이 추가됐나?" 0.05초마다 확인
    ▼
[2단계] OS 버퍼 강제 밀어내기
    │  LLEAP가 아직 파일에 안 쓴 데이터를 강제로 디스크에 기록시킴
    ▼
[3단계] 데이터 파싱 (읽기 + 분류)
    │  XML 텍스트에서 "시나리오 시작", "CPR", "약물 투여" 등을 뽑아냄
    ▼
[4단계] 외부 전송
    │  SIMPREC 서버에 HTTP로 전송 + 웹 대시보드에 WebSocket으로 실시간 표시
    ▼
SIMPREC 녹화 시스템 / 웹 대시보드
```

---

### [1단계] 로그 파일 감시

**비유:** 택배 추적 — 택배함을 0.05초마다 열어보고, 새 택배가 있으면 꺼내는 것

**동작 원리:**

LLEAP 시뮬레이터는 내부적으로 WCF라는 통신 방식을 사용하는데, 이 통신 내용이 `client_wcf.svclog`라는 XML 로그 파일에 자동으로 기록됩니다.

SimMonitor는 이 파일을 **0.05초(50ms) 간격으로 계속 확인**합니다. "마지막으로 읽은 위치"를 기억해 두고, 그 이후에 추가된 부분만 읽습니다.

```
파일 전체:  [이미 읽은 부분 ████████████][새로 추가된 부분 ████]
                                          ↑
                                   여기부터만 읽음
```

**왜 0.05초인가?**
- 너무 빠르면 CPU를 낭비하고, 너무 느리면 이벤트 감지가 늦어짐
- 0.05초는 사람이 느끼기엔 "즉시"이면서, 시스템 부하는 거의 없는 적정값

---

### [2단계] OS 버퍼 강제 밀어내기 (ForceFlush)

**비유:** 편지를 쓰는 사람(LLEAP)이 아직 봉투에 안 넣은 편지를 강제로 우체통에 넣게 만드는 것

**동작 원리:**

LLEAP가 로그를 쓸 때, 운영체제(Windows)가 효율을 위해 **메모리에 임시 저장(버퍼링)** 해둡니다. 즉, LLEAP가 "CPR 시작"이라고 기록해도 실제 파일에는 아직 안 써진 상태가 수십 초간 지속될 수 있습니다.

```
LLEAP가 기록 → [OS 메모리 버퍼] ──(20~75초 후)──→ 실제 파일
                                                    ↑
                               이때까지 SimMonitor는 볼 수 없음
```

SimMonitor는 이 문제를 **Windows 운영체제 명령(FlushFileBuffers)**으로 해결합니다.
0.2초마다 OS에게 "지금 버퍼에 있는 거 전부 파일에 써라"고 명령합니다.

```
LLEAP가 기록 → [OS 메모리 버퍼] ──(강제 밀어내기!)──→ 실제 파일
                   ↑                                    ↑
           SimMonitor가 0.2초마다               이제 SimMonitor가
           "지금 당장 써!" 명령                  즉시 읽을 수 있음
```

**결과:** 원래 20~75초 걸리던 지연이 **0.5초 미만**으로 단축

---

### [3단계] 데이터 파싱 (읽기 + 분류)

**비유:** 신문에서 키워드를 찾아 형광펜으로 표시하는 것

**동작 원리:**

읽어온 XML 텍스트에서 **12가지 이상의 패턴**을 검색해서 이벤트를 분류합니다.

```
XML 텍스트에서 찾는 것          →  분류 결과
─────────────────────────────────────────────────
"startScenario" 가 포함되어 있으면  →  세션 시작 (SESSION)
"stopScenario" 가 포함되어 있으면   →  세션 종료 (SESSION)
"eventOccurred" 가 포함되어 있으면  →  임상 이벤트 (EVENT) — CPR, 기관삽관 등
"drugEventOccurred" 가 포함되어 있으면 → 약물 투여 (DRUG)
"SetActiveState" 가 포함되어 있으면  →  상태 전환 (STATE)
"HeartRate", "SpO2" 등이 포함되면   →  바이탈 사인 (PARAM)
... 등등
```

추출된 데이터는 **한글로 번역**됩니다 (454개 파라미터 번역 테이블 보유).

```
영문 원본:    "Laerdal.Event.CPRStarted"
한글 번역:    "CPR 시작"

영문 원본:    "EcgHeartRate_bpm = 120"
한글 번역:    "심박수 = 120 bpm"
```

**중복 제거도 이 단계에서 수행:**

같은 이벤트가 여러 경로로 중복 감지되는 것을 방지하기 위해, 4단계에 걸쳐 필터링합니다.

```
필터 1: 같은 XML 블록을 두 번 읽은 건 아닌지? (고유 ID로 확인)
필터 2: 같은 밀리초에 같은 이벤트가 2번 나온 건 아닌지?
필터 3: 0.1초 이내에 완전히 동일한 이벤트가 또 왔는지?
필터 4: 1초 이내에 같은 파라미터가 같은 값으로 또 왔는지?

→ 진짜 새 이벤트만 통과
→ 단, CPR 10회 연속 시행 같은 의도적 반복은 정상 통과
```

---

### [4단계] 외부 전송

**비유:** 분류된 이벤트를 두 곳에 동시에 배달하는 것

**동작 원리:**

파싱된 이벤트는 **두 가지 경로**로 동시에 전송됩니다.

```
파싱된 이벤트
    ├──→ SIMPREC 서버 (HTTP REST API)
    │      ├── 세션 시작: POST /sessions/start  (녹화 시작 트리거)
    │      ├── 세션 종료: POST /sessions/stop   (녹화 종료 + 영상 저장)
    │      ├── 이벤트:   POST /events           (CPR, 약물 등 기록)
    │      └── 바이탈:   POST /vitals           (HR, SpO2, BP 등)
    │
    └──→ 웹 대시보드 (WebSocket, 포트 8080)
           └── 브라우저에서 실시간 바이탈/이벤트 표시
```

**이벤트 전송의 핵심: "쏘고 잊기(Fire-and-Forget)" 패턴**

바이탈이나 임상 이벤트를 SIMPREC에 보낼 때, 응답을 기다리지 않고 바로 다음 작업을 합니다. 왜냐하면 응답을 기다리면 그 사이에 새 이벤트가 밀려서 지연이 생기기 때문입니다.

```
일반적인 방식:  이벤트 전송 → 응답 대기(0.5초) → 다음 이벤트 처리
                                ↑ 이 시간 동안 새 이벤트가 밀림

SimMonitor 방식: 이벤트 전송(보내기만 함) → 즉시 다음 이벤트 처리
                 └── 응답은 백그라운드에서 알아서 처리
```

단, **세션 시작/종료는 응답을 기다립니다** (녹화 시작 확인이 필수이므로).

## 3. 트러블슈팅 논리 — "왜 tshark를 쓰려다 말았는가"

### 배경: 문제 상황

LLEAP 로그 파일에 이벤트가 기록되기까지 **20~75초 지연**이 있었습니다. 이 때문에 녹화가 20초 늦게 시작되거나, 종료 신호를 1분 넘게 못 받는 문제가 발생했습니다.

### 초기 시도: tshark(네트워크 패킷 캡처) 도입

"파일이 늦게 기록되니까, 파일을 거치지 않고 **네트워크 통신을 직접 가로채자**"는 접근이었습니다.

```
기존 방식:  LLEAP → 로그 파일(20~75초 지연) → SimMonitor
tshark 방식: LLEAP → 네트워크 패킷(즉시) → tshark → SimMonitor
```

tshark는 Wireshark의 명령줄 버전으로, 네트워크 패킷을 실시간 캡처할 수 있습니다. 실제로 이벤트를 **밀리초 단위**로 감지하는 데 성공했습니다.

### tshark를 비활성화한 3가지 이유

#### 이유 1: 관리자 권한 필수

```
tshark는 네트워크 인터페이스에 직접 접근하므로 Windows 관리자 권한이 필수입니다.

문제: 교육 환경의 PC는 일반 사용자 계정으로 운영되는 경우가 많음
     → 관리자 권한을 요구하면 배포·운영이 어려워짐
     → IT 관리자의 별도 승인 절차 필요
```

#### 이유 2: SimMan의 TLS 암호화 비호환

```
SimBaby:  Port 7558, 평문 통신 → tshark로 내용을 읽을 수 있음 ✅
SimMan:   Port 15021, TLS 암호화 통신 → tshark로 내용을 읽을 수 없음 ❌

SimMonitor는 SimBaby와 SimMan을 모두 지원해야 하는데,
tshark 방식은 SimMan에서 작동하지 않으므로 범용 솔루션이 될 수 없었습니다.
```

#### 이유 3: 코드 복잡성 폭증 (듀얼 파이프라인)

```
tshark와 WCF 로그 양쪽에서 동일한 이벤트가 감지되므로,
"같은 이벤트를 2번 보내지 않기" 위한 중복 제거 로직이 필요했습니다.

추가된 코드:
- SimulatorMonitor.cs에 #if USE_TSHARK 블록 30개
- WcfEventCapture.cs에 #if USE_TSHARK 블록 12개
- 카운트 기반 크로스파이프라인 중복 제거 엔진

이 복잡성은 유지보수 비용을 크게 높이고, 새로운 버그를 만들 가능성을 높였습니다.
```

### 최종 판단

```
tshark 접근법:  지연 해결 ✅ | 관리자 권한 ❌ | SimMan 지원 ❌ | 유지보수 ❌
WCF 최적화:     지연 해결 ✅ | 관리자 권한 ✅ | SimMan 지원 ✅ | 유지보수 ✅

→ WCF 파이프라인 자체를 최적화하는 것이 올바른 선택
→ tshark 코드는 삭제하지 않고 조건부 컴파일로 보존 (필요 시 활성화 가능)
```

## 4. 녹화 조기 종료 방지 — 3중 방어 체계

### 문제 상황

LLEAP 시뮬레이터가 시나리오를 시작할 때, **내부 초기화 과정에서 "시작"과 "종료"를 동시에 기록**하는 비정상적 동작이 있었습니다.

```
LLEAP 내부 동작 (시나리오 시작 시):
  1. "시나리오 시작" 기록  ← 정상
  2. 핸드셰이크 과정에서 "시나리오 종료" 기록  ← 가짜 종료!
  3. 실제 시나리오 진행 시작

SimMonitor 입장:
  T+0초:  "시작" 감지 → SIMPREC 녹화 시작 ✅
  T+25초: "종료" 도착 (가짜! WCF 버퍼 지연으로 뒤늦게 도착)
          → SIMPREC 녹화 종료 ❌ (교수자 모르게 녹화 중단!)
```

### 3중 방어 체계

```
가짜 "종료" 신호가 도착
    │
    ▼
[방어 1] Grace Period (10초)
    "시작한 지 10초도 안 됐으면 가짜 종료로 판단"
    │
    │  10초 이내 → 차단 ✅ (무시한 횟수를 기억해 둠)
    │  10초 이후 → 다음 방어로 이동 ▼
    │
[방어 2] earlyStop 카운트
    "방어 1에서 막은 횟수만큼, 이후에도 추가로 막음"
    │
    │  예시: 방어 1에서 2개 막았으면, 이후 2개도 추가로 막음
    │  (WCF 버퍼 지연으로 같은 가짜 종료가 늦게 또 올 수 있으므로)
    │
    │  아직 막을 여유 있음 → 차단 ✅
    │  모두 소진됨 → 다음 방어로 이동 ▼
    │
[방어 3] 최소 녹화 시간 (10초)
    "녹화 시작 후 10초가 안 됐으면 무조건 차단"
    │
    │  10초 미만 → 차단 ✅
    │  10초 이상 → 진짜 종료로 판단 → 녹화 종료 처리
    │
    ▼
정상 종료 처리
```

### 왜 3중인가?

```
방어 1만 있으면: 10초 이후에 도착하는 지연된 가짜 종료를 못 막음
방어 2만 있으면: 최초 시나리오에서는 카운트가 0이라 아무것도 못 막음
방어 3만 있으면: 실제로 10초 후에 진짜 종료한 경우를 구분 못 함

→ 3개가 겹쳐야 모든 케이스를 커버
```

## 5. 세션 데이터 격리 — Zombie Data 방지

### 문제 상황

시나리오를 종료하고 다시 시작하면, **이전 세션의 데이터가 새 세션에 섞여 나오는** 현상.

```
세션 1: 심박수 120 → 종료
세션 2: 심박수 120으로 시작
        → SimMonitor: "120→120이니까 변화 없음, 안 보내도 됨" (!)
        → SIMPREC에 심박수 데이터 누락
```

### 해결 원리

```
세션 종료 시:
  ├── "마지막으로 본 바이탈" 기억을 지움 (변경 감지 캐시 초기화)
  │    → 다음 세션에서 120이 와도 "새 값"으로 인식
  │
  ├── 이벤트 히스토리를 지움
  │    → 이전 세션 이벤트가 새 세션에 표시되지 않음
  │
  └── 중복 제거 테이블을 지움
       → 이전 세션의 이벤트 ID가 새 세션의 같은 이벤트를 차단하지 않음

주의: "현재 표시 중인 바이탈 값"은 지우지 않음!
  → LLEAP는 변경된 파라미터만 보내므로, 표시값까지 지우면
    아직 안 온 파라미터가 "--"로 표시되는 문제 발생
```

## 6. 이벤트 버퍼링 — 녹화 시작 전 이벤트 손실 방지

### 문제 상황

```
T+0.0초: "시나리오 시작" 감지 → SIMPREC에 "녹화 시작" API 호출 (응답 대기 중...)
T+0.3초: "상태: All normal" 이벤트 도착
T+0.5초: "약물 투여" 이벤트 도착
T+2.0초: SIMPREC "녹화 시작" 응답 도착 (session_id 확보)

→ 0.3초와 0.5초에 도착한 이벤트는 어디로?
  session_id가 없어서 SIMPREC에 보낼 수 없었음 → 손실!
```

### 해결 원리

```
T+0.0초: "녹화 시작 대기 중" 플래그 ON
         → 이후 도착하는 이벤트는 대기열(큐)에 임시 저장

T+0.3초: "상태: All normal" → 큐에 저장
T+0.5초: "약물 투여" → 큐에 저장

T+2.0초: SIMPREC 응답 도착, session_id 확보
         → "녹화 시작 대기 중" 플래그 OFF
         → 큐에 있던 이벤트를 순서대로 SIMPREC에 전송
         → 이후 이벤트는 즉시 전송

결과: 이벤트 손실 0건
```

## 7. 면접 대비 — 예상 질문과 답변

### Q: "이 프로젝트에서 가장 어려웠던 기술적 문제는?"

> LLEAP 시뮬레이터의 WCF 로그 파일이 OS 버퍼링 때문에 20~75초 지연되는 문제였습니다. 처음에는 tshark(네트워크 패킷 캡처)로 우회했지만, 관리자 권한 필수·TLS 비호환·코드 복잡성 문제로 운영 환경에서 쓸 수 없었습니다. 최종적으로 Windows의 FlushFileBuffers API로 OS 버퍼를 200ms마다 강제 플러시하는 방식으로 해결하여, 외부 의존성 없이 지연을 0.5초 미만으로 줄였습니다.

### Q: "왜 tshark를 포기하고 WCF 최적화를 선택했나?"

> 세 가지 이유입니다. 첫째, tshark는 관리자 권한이 필수인데 교육 환경 PC는 일반 사용자 계정으로 운영됩니다. 둘째, SimMan 시뮬레이터는 TLS 암호화 통신을 사용하여 tshark로 내용을 읽을 수 없습니다. 셋째, 두 파이프라인의 중복 이벤트를 처리하는 로직이 코드 복잡도를 크게 높여 유지보수가 어려웠습니다. WCF 파이프라인 자체를 최적화하면 이 세 문제가 모두 해결되므로 더 나은 선택이었습니다.

### Q: "녹화 조기 종료는 왜 3중으로 방어하나?"

> LLEAP가 시나리오 시작 시 가짜 종료 신호를 보내는데, WCF 버퍼 지연 때문에 이 신호가 10초 이후에도 도착할 수 있습니다. 시간 기반 필터(Grace Period) 하나로는 지연 도착을 못 막고, 카운트 기반 필터 하나로는 최초 시나리오를 못 막습니다. 세 가지 필터가 각각 다른 케이스를 커버하여, 어떤 타이밍으로 가짜 신호가 와도 차단되면서 진짜 종료는 정상 처리됩니다.

### Q: "시스템 아키텍처를 한마디로 설명해 준다면?"

> WCF 로그 파일 기반의 이벤트 드리븐 미들웨어입니다. LLEAP가 기록하는 WCF 서비스 로그를 50ms 간격으로 감시하고, 12가지 정규식 패턴으로 이벤트를 분류한 뒤, 4단계 중복 제거를 거쳐 SIMPREC REST API와 WebSocket 대시보드로 실시간 전송합니다.
